High-Speed 64-bit Multiplier Architecture

A three-stage integer multiplier that combines odd-multiple precomputation, nibble-wide crossbar selection, and hierarchical accumulation to keep throughput high without dragging down clock speed.

16 Nibble-selected multiples drive the partial-product generation path instead of a deeper bit-level field.
3 stages Balanced pipeline from operand conditioning to final writeback result.
5 ops MUL, MULH, MULHSU, MULHU, and MULW all share the same datapath.

Multipliers Usually Become the Clock-Limiting Block

In a 64-bit execution pipeline, the multiplier is often the longest combinational path in the room. Iterative designs protect area but cost throughput. More conventional radix-4 style approaches reduce the work, but they still leave a relatively large reduction network and plenty of routing pressure.

The target here was different: keep the datapath regular, preserve high throughput, and support the full set of RV64 multiply result formats without collapsing timing around a monolithic final adder. The design therefore pushes work into deliberate stages: precompute only the hard odd coefficients, select directly by nibble, then reduce locally before the final 128-bit collapse.

Odd Multiples First, Local Sums Before Global Sums

The final RTL behaves like a radix-16 multiplier, but the implementation leans on direct multiple selection rather than recoding-heavy control logic. The result is a more regular structure that maps cleanly to a pipelined execution stage.

01

Operand conditioning. The front end derives negative, upper, and word control, converts signed cases into magnitude form where required, and sign-extends 32-bit inputs for MULW.

02

Odd-multiple precomputation. Stage 1 generates 1x, 3x, 5x, 7x, 9x, 11x, 13x, and 15x. Even multiples are not separately built here; they are derived later as shifts of the stored odd values.

03

Nibble-wide crossbar selection. Stage 2 expands the full 0x to 15x set and uses each 4-bit multiplier nibble as a direct select into a 16-output crossbar. One nibble, one chosen multiple, one fixed 4-bit alignment step.

04

Local accumulation. The selected rows are not thrown into one huge adder at once. Instead, four neighboring groups are collapsed into four 80-bit intermediate sums, which keeps routing more local and reduces middle-stage sprawl.

05

Final reduction and result shaping. Stage 3 widens to 128 bits, restores sign if needed, selects either the upper or lower half of the product, and tags the writeback as word or doubleword.

The Source Matches the Story

The implementation is explicit about its stage boundaries and arithmetic structure. The control bits move through q0, q1, and q2 alongside the datapath, and the code exposes exactly where the odd-multiple bank, crossbar expansion, and final accumulation happen.

Stage 1

always_comb res_0001 = multiplicand;
always_comb res_0011 = {res_0001, 1'b0} + res_0001;
always_comb res_0101 = {res_0001, 2'b0} + res_0001;
always_comb res_0111 = {res_0001, 2'b0} + res_0011;

The hard work starts by generating only the odd coefficients that are expensive to recover later.

Stage 2

always_comb ext68_res['b1010] = {res_0101_q0, 1'b0};
xbar #(.NUM_INPUT(16), .NUM_OUTPUT(16), .DATA_WIDTH(68)) u_xbar (...);
res80[i] = {ext80[4*i+3], 12'b0} + {ext80[4*i+2], 8'b0} +
           {ext80[4*i+1], 4'b0} + ext80[4*i];

The middle stage reconstructs the full 16-entry set, performs nibble-based selection, then compresses four adjacent results at a time.

Stage 3

always_comb final_sum =
  {res128[3], 48'b0} + {res128[2], 32'b0} +
  {res128[1], 16'b0} + res128[0];
always_comb semi_final_res = negative_q1 ? ~final_sum + 1 : final_sum;
always_comb final_res = upper_q1 ? semi_final_res[127:64] : semi_final_res[63:0];

The final stage stays simple because most of the entropy was already squeezed out in the local accumulation step.

Nibble Decomposition

The 64-bit multiplier operand is consumed as sixteen 4-bit selects. That gives radix-16 style behavior with direct mux selection instead of a signed Booth recoder.

Routing Discipline

Four local sums absorb nearby nibble contributions before the last adder ever sees them, which helps contain long wires and keeps the stage split physically sane.

Built for RV64 Multiply Semantics, Not Just a Standalone Demo

The block is wired around real execution-stage needs. Signedness handling happens before the multiply bank, upper selects high-half variants, word controls 32-bit result shaping, and the writeback side emits the corresponding size metadata for the downstream pipeline.

Validation Detail

The accompanying randomized testbench checks every accepted transaction against a reference multiplication model, including sign restoration, high-half selection, and word-mode truncation. It is structured to wait for 100000 accepted operations of each multiply variant before concluding.

A Fast, Regular Datapath Without Recoder Sprawl

This multiplier architecture trades a little more front-end structure for a much cleaner pipeline split. The odd-multiple bank is predictable, the crossbar mapping is direct, and the reduction tree is intentionally staged instead of left to become one late critical path.

That combination makes it a practical choice when sustained throughput and timing closure matter more than chasing the smallest possible control block. It is not just mathematically correct; it is organized so the RTL, the pipeline boundaries, and the instruction semantics all reinforce one another.
