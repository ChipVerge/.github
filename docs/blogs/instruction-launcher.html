SystemVerilog Scheduling Hazard Control

Hazard-Aware Instruction Launcher

A decoded-instruction launch queue that keeps execution fed while screening register hazards, serializing memory traffic when required, and arbitrating eligible instructions through a clean ready-valid interface.

8 slots The default build exposes an eight-entry launch window because NUM_OUTSTANDING is seven and the queue arrays span 0 through NOS.
64 lock bits Hazard checking scales across the full integer and floating-point register namespace through a propagated lock vector.
1 launch port All eligible slots collapse to one selected decoded instruction and one valid signal at the execution boundary.

Keep the Execution Units Busy Without Launching the Wrong Instruction

A frontend pipeline can decode instructions faster than the backend can safely consume them. The hard part is not simply buffering decoded work. The hard part is deciding when each instruction is actually safe to issue, especially when destination registers are already locked, blocking instructions must fence younger work, or memory operations need ordering pressure that pure register scoreboarding would miss.

This launcher solves that control problem as a standalone RTL block. It holds multiple decoded instructions in flight, re-checks them against live lock information, derives request eligibility per slot, and only exposes one chosen launch candidate when the downstream execution stage is ready.

Queue First, Check Every Slot, Then Arbitrate

The implementation uses a split-pipeline structure rather than a monolithic FIFO plus sideband scoreboard. That keeps the storage, hazard evaluation, and launch arbitration all expressed in regular, synthesizable stages.

01

Instruction staging. The incoming decoded instruction is attached to pl_ins[0] and then fanned through a chain of pipeline_split stages, creating a resident launch window instead of a single-cycle pass-through path.

02

Hazard propagation. A reg_gnt_ckr instance is attached to every resident slot. It examines reg_req, rd, blocking, and mem_op against propagated locks and mem_busy state, then emits that slot's launch request.

03

Launch arbitration. All slot requests feed a fixed-priority arbiter, gated by instr_out_ready_i. An encoder translates the one-hot grant into gnt_idx, which directly selects the outgoing decoded instruction.

04

Flush and drain behavior. A synchronous clear_i path exists to reset the resident queue state, while the clear chain also ensures launch activity removes the selected instruction instead of collapsing the whole window.

05

Downstream contract. The module emits exactly one decoded_instr_t plus a valid bit, making it easy to sit between decode and execution without forcing the backend to understand the entire queue state.

The Source Exposes the Scheduling Strategy Directly

The interesting part of this design is that the launch policy is not buried in opaque control logic. The queue topology, the per-slot checks, and the arbitration path are all easy to point to in the RTL.

Queue Topology

assign pl_ins[0]        = instr_in_i;
assign pl_ins_valid[0]  = instr_in_valid_i;
assign instr_in_ready_o = pl_ins_ready[0];

for (genvar i = 0; i < NOS; i++) begin : g_splits
  pipeline_split #( .DW($bits(instr_in_i)) ) u_pipeline_split (...);
end

The instruction enters a staged launch window rather than an unstructured holding register.

Hazard Checking

reg_gnt_ckr #() u_reg_gnt_ckr (
  .pl_valid_i(pl_outs_valid[i]),
  .blocking_i(pl_outs[i].blocking),
  .rd_i(pl_outs[i].rd),
  .reg_req_i(pl_outs[i].reg_req),
  .locks_i(locks[i]),
  .mem_busy_i(mem_busy[i]),
  .arb_req_o(arb_req[i])
);

Every resident slot gets its own eligibility decision based on the decoded dependency footprint.

Launch Selection

fixed_priority_arbiter #( .NUM_REQ(NOS + 1) ) u_fixed_priority_arbiter (...);
encoder #( .NUM_WIRE(NOS + 1) ) u_encoder (
  .wire_in(arb_gnt),
  .index_o(gnt_idx),
  .index_valid_o(instr_out_valid_o)
);

assign instr_out_o = pl_outs[gnt_idx];

Grant generation and slot selection are separated cleanly, which keeps the output path explicit and debuggable.

Blocking Semantics

The design treats blocking instructions as more than ordinary dependency checks. Their presence can suppress younger work from launching past them, which is essential for correctness around serialized operations.

Memory Awareness

The propagated mem_busy chain means the launcher does not rely on register hazards alone. Memory operations can also influence whether younger residents are eligible to issue.

Stress-Tested Against a Behavioral Launch Model

The accompanying testbench does more than toggle handshakes. It builds a software-style model of the resident pipeline, randomizes lock profiles, incoming decoded instructions, downstream readiness, and blocking or memory attributes, then compares the expected launched instruction against the RTL output.

Testbench Detail

Reset behavior is checked explicitly, the driver randomizes valid and ready conditions at runtime, and the monitor runs until either the expected launch comparisons complete cleanly or a fatal timeout trips after 150001 clock edges.

A Focused Dispatch Block for Real Backend Pressure

This design turns instruction launch into a well-bounded RTL problem: store several decoded operations, evaluate their hazard state continuously, and dispatch exactly one correct candidate when the backend is ready. The structure stays regular, the interface remains simple, and the launch policy is visible in the source instead of being hidden in a large hand-written controller.

That makes the block useful anywhere a decode stage needs a disciplined handoff into execution without giving up throughput or correctness under scoreboard pressure.
